## Background the executor needs (read once)

How shadcn serves multiple styles (confirmed from shadcn docs, CLI v4):

- A registry URL may contain a `{style}` placeholder, e.g.
  `https://audio-ui.xyz/r/{style}/{name}.json`. When a consumer runs
  `npx shadcn@latest add @audio/player`, the CLI substitutes **their** `style`
  (from their `components.json`, e.g. `base-nova`) → fetches
  `https://audio-ui.xyz/r/base-nova/player.json`.
- Source is conventionally laid out as `registry/[STYLE]/[NAME]`. This repo currently
  has a single `src/registry/default/` tree (= the `base-luma` look) and serves a flat
  `https://audio-ui.xyz/r/{name}.json`.
- The audio components mostly use **semantic tokens** (`bg-primary`, `var(--radius)`)
  that already adapt to whatever theme/preset the consumer initialized, and they pull
  their base primitives from official `@shadcn/*` items (which already match the
  consumer's preset). So "supporting other styles" is primarily a **packaging/serving**
  change plus **optional per-style restyling**, not a rewrite.

**Chosen architecture (why no mass rename):** `src/registry/default/` stays the
canonical source. The build becomes style-aware and emits `public/r/<style>/<name>.json`
for every style in a `STYLES` list. A style gets a genuinely different variant only by
adding override files under `src/registry/<style>/...`; anything not overridden falls
back to `default/`. This avoids rewriting the ~150 `@/registry/default/...` import
sites the docs app depends on, and lets variants be added incrementally.

## Dependency notes

- **001 is independent** of 002/003 — it only removes an unused `radix-ui` dependency
  and an unused brand icon. Land it first or in parallel; it's the smallest unit.
- **003 requires 002** — 003 authors per-style override files and a docs preview

- **004 depends on 002** — the `/create` builder's style picker drives which per-style

## Findings considered and rejected

- **Full `default/` → `base-luma/` directory rename**: technically the "purest" match
  for shadcn's `[STYLE]/[NAME]` convention, but it churns ~150 import sites across the
  docs app for zero functional gain over the override-with-fallback approach. Rejected
  in favor of keeping `default/` as the canonical source (see 002). Revisit only if the
  team later wants the source tree to literally mirror style ids.
- **Per-style duplication of every component (6×7 files)**: rejected as a maintenance
  trap. The override-with-fallback model in 002 means a style only carries the files
  that actually differ.
- **Shipping a `registry:base` whole-design-system item**: out of scope. audio/ui ships
  audio components on top of the consumer's existing shadcn setup; it should not try to
  own the consumer's base/theme/fonts.
